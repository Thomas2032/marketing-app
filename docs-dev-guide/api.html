4. API 路由與企劃狀態機

後端 FastAPI 路由接口設計與 Campaign 行銷企劃的核心狀態轉移流程。

REST API 路由清單

方法 (Method) 端點 (Endpoint) 請求載荷 (Payload) 狀態動作 / 業務邏輯 回傳結構 (Response)
POST /api/v1/campaigns CampaignCreate 於庫中建立新企劃，初始狀態設為 draft，初始化對話歷史 `messages`。 CampaignRead
GET /api/v1/campaigns Query: user_id, limit 列出該用戶的所有企劃摘要（不包含 outputs 明細），依更新時間倒序排列。 list[CampaignSummaryRead]
GET /api/v1/campaigns/{id} 無 讀取特定企劃詳情，並**級聯加載 (selectinload)** 所有關聯的 `outputs`。 CampaignRead
POST /api/v1/campaigns/{id}/run 無 若使用佇列 (`use_queue=True`)，狀態轉為 queued 並入 arq 佇列；否則轉為 running 並觸發 BackgroundTasks 後台執行緒。 CampaignRunResponse
POST /api/v1/campaigns/{id}/messages { content: string } 將用戶追問訊息寫入 `messages` 中，**重置並清空 `tool_calls` 與 `last_completed_tool`**，並將狀態設為 `running` / `queued` 重新跑一遍 LangGraph。 CampaignRead

企劃狀態遷移圖 (Campaign Status Transitions)

DRAFT 剛建立 (POST)
➔ POST /run
QUEUED 若 use_queue = True
⬇ arq 工作進程接單
RUNNING 正在執行 LangGraph
➔
COMPLETED 審核通過 (Approved)
REVIEW 審核有修改建議
POST /messages ➔ 重置並回到 RUNNING
FAILED 執行異常中斷

追問回饋機制設計 (Continuous Chat Rerun Loop)

當用戶對生成文案不滿意並輸入微調要求時，前端會調用 POST /messages 端點。後端並非重新建立一個企劃，而是採取以下清空斷點與變更歷史的步驟來進行重新運行 (Rerun)： 

# 於 api/routes/campaigns.py:post_campaign_message
current_state = dict(campaign.state)

# 1. 追加入對話歷史 (包含 role: "user" 與微調文字)
current_state["messages"].append({
    "role": "user",
    "content": payload.content,
    "created_at": datetime.utcnow().isoformat()
})

# 2. 清空先前的工具呼叫歷史，讓前端進度條重置
current_state["tool_calls"] = []

# 3. 清空斷點指示，強迫圖執行從第一個節點 (extractor) 重新開始
current_state["last_completed_tool"] = None

campaign.state = current_state
campaign.status = CampaignStatus.RUNNING
await db.commit()

# 4. 再次觸發後台執行
background_tasks.add_task(execute_campaign, str(campaign_id))
